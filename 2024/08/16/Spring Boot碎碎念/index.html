业务层开发(Service)

数据层开发告一段落，下面进行业务层开发，其实标准业务层开发很多初学者认为就是调用数据层，怎么说呢？这个理解是没有大问题的，更精准的说法应该是组织业务逻辑功能，并根据业务需求，对数据持久层发起调用。有什么差别呢？目标是为了组织出符合需求的业务逻辑功能，至于调不调用数据层还真不好说，有需求就调用，没有需求就不调用。

一个常识性的知识普及一下，业务层(Service)的方法名定义一定要与业务有关，例如登录操作

而数据层(Mapper)的方法名定义一定与业务无关，是一定，不是可能，也不是有可能，例如根据用户名密码查询

依赖包相关

朦朦胧胧中感觉starter与parent好像都是帮助我们简化配置的，但是功能又不一样，梳理一下。

starter是一个坐标中定了若干个坐标，以前写多个的，现在写一个，是用来减少依赖配置的书写量的。

parent是定义了几百个依赖版本号，以前写依赖需要自己手工控制版本，现在由SpringBoot统一管理，这样就不存在版本冲突了，是用来减少依赖冲突的。

实际开发应用方式

实际开发中如果需要用什么技术，先去找有没有这个技术对应的starter

如果有对应的starter，直接写starter，而且无需指定版本，版本由parent提供
如果没有对应的starter，手写坐标即可
实际开发中如果发现坐标出现了冲突现象，确认你要使用的可行的版本号，使用手工书写的方式添加对应依赖，覆盖SpringBoot提供给我们的配置管理

方式一：直接写坐标
方式二：覆盖中定义的版本号，就是下面这堆东西了，哪个冲突了覆盖哪个就OK了

Redis

数据库和缓存不一致如何解决？

在操作数据库后删除缓存。

为什么不选择更新缓存？

更新数据库的次数 = 更新缓存次数 >= 删除缓存次数（没缓存的时候不需要删除）

按照这个逻辑，多次更新缓存时，中间若是没有查询请求便没有意义，属于无效操作

因此，选择删除缓存。

为什么先操作（增删改）数据库，后删除缓存？

操作数据库的时间相对于删除缓存是比较长的，

如果先删除缓存，后操作数据库，那么会有一种情况是，在操作数据库的途中，也就是事务还没有提交的时候，查询操作会建立了缓存，

基于MySQL默认情况下可重复读的事务隔离级别，这个数据是事务提交前的数据，也就是说这个缓存是过时的，是不准确的，会导致后

续的查询都拿到这个错误结果。

如果先操作数据库，后删除缓存，拿到错误缓存的概率就变得非常小了，拿到错误缓存的例子：在提交事务后，删除缓存前的这个时间段

刚好有查询操作会拿到错误缓存，但这个概率很小，并且在下一次查询的时候（删除缓存的操作完成）往往会拿到正确的缓存。

缓存穿透是什么，有什么应对措施吗？

缓存穿透：缓存和数据库中都不存在请求中的数据，导致大量请求直接访问数据库。

解决方法：缓存空对象，布隆过滤器，加强用户权限校验，做好数据的基础格式校验（在增强id的复杂度，避免被猜测id规律的基础

上），做好热点参数的限流

缓存空对象：简单，但占用内存，且可能的缓存和数据库的不一致（一开始查询昵称为a的用户没有找到，后面有人改名为a，会拿到空缓

存，从而发生不一致）

布隆过滤器：不占内存，没有多余key，但实现复杂，且有误判可能（布隆过滤器的实现是基于哈希思想的，必定存在哈希冲突）

缓存雪崩呢？

缓存雪崩：同一时间段内大量缓存key失效(a) 或 Redis服务宕机(b)，导致大量请求到达数据库。

解决方案：a:给不同的key添加随机的TTL（设置波动时间，不让它在短时间内全部失效），给缓存业务添加降级限流策略(cloud安全)，给

业务添加多级缓存(nginx、Redis、JVM)
b: 使用 Redis 集群，防止宕机

缓存击穿？

缓存击穿：热点数据key，也就是被高并发访问并且缓存重建业务比较复杂的key失效，导致大量请求越过缓存访问数据库，给数据库带来

巨大压力

解决方案：

互斥锁（第一个没有命中缓存的请求线程获取锁，其余阻塞，直到第一个请求写入缓存后，其余请求从该缓存中获取数据），可能导致

死锁

逻辑过期（会有短期的不一致），具体实现：每次请求都检查过期时间，第一个发现过期的拿锁，并开一个新进程去重建缓存(包括刷

新逻辑TTL)，之后返回过期的缓存，后续发现过期的进程拿不到锁直接返回过期数据，再后面不过期直接返回数据。 30min
